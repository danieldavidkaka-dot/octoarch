# Contributing to OctoArch 🐙

Thank you for your interest in improving OctoArch! This project was born with the vision of creating a local, modular, and stateful Cognitive Runtime highly capable of integrating with any system via the MCP protocol, native tools, Multi-Model Orchestration, and Autonomous Swarm Intelligence.

With the release of v5.0+, OctoArch strictly follows Clean Architecture principles, enterprise-grade security standards, and features a revolutionary Auto-Programming pipeline (Hot-Swapping) backed by Infinite Cloud Memory.

## 🤝 How can you help?

Currently, the core cognitive engine is highly stable. The areas where the community can add the most value are:

* **Swarm Intelligence & Multi-Model Prompts:** Improving the Sub-Agent (Worker) prompt engineering in `src/swarm/worker.ts` to make the AI forge even more complex TypeScript tools. You can also help refine the delegation logic so OctoArch perfectly routes heavy reasoning tasks to specialized models like DeepSeek.
* **Dynamic Skills (Zero-Code):** You don't need to know TypeScript to contribute! Write highly-optimized Markdown manuals (`.md`) teaching the AI specific frameworks or workflows, and submit them to the `workspace/skills/` directory.
* **MCP (Model Context Protocol) Servers:** If you build an MCP server to connect OctoArch with Notion, Spotify, Home Assistant, SQL databases, or commercial ERPs (like SAP/Profit Plus), please open a PR to add it to our recommended integrations list.
* **Native Tools vs. Dynamic Tools:** Build new core system plugins in the `src/tools/` directory (e.g., native PDF generation, advanced WebSockets). *Note: Simple API wrappers or data parsers should ideally be auto-generated by the AI Worker into `src/dynamic_tools/` rather than hardcoded.*
* **Cognitive Loop & Cloud Memory:** Improvements in reasoning, context compression, and PostgreSQL (Supabase) state synchronization within `src/core/session_manager.ts` and `src/core/llm.ts`.

## 🛠️ Development Guidelines

### 1. Clean Architecture (No "God Classes")
Use strict TypeScript. Keep functions small and modular. Do not overload `llm.ts` with execution logic. Delegations must pass through `tool_orchestrator.ts`. Do not create random `.ts` or `.txt` files for core prompts; always use the `PromptManager`. If extending the Swarm, isolate the logic within the `src/swarm/` directory and utilize the `src/temp_tools/` sandbox for compilations.

### 2. Security First (Zero-Trust & Enterprise-Grade)
* **Data Ingestion (CRITICAL):** If your tool receives files from external sources (WhatsApp, Gmail, HTTP), you MUST NEVER trust the provided mimetype or extension. You must pass the raw buffer through `FileValidator.validateBuffer(buffer, name)` to verify its "Magic Numbers".
* **Worker Secrets & API Keys:** Sub-Agents are strictly prohibited from hardcoding secrets. Any AI-generated tool that connects to external APIs MUST consume credentials exclusively from the `env.WORKER_SECRETS` payload (validated via Zod in `src/config/env.ts`).
* **Terminal & Swarm Worker:** Shell execution (`ShellTool`) MUST use a strict AllowList. Never allow commands that can easily exfiltrate data. Furthermore, the autonomous coding worker (`worker.ts`) must compile code safely in the `src/temp_tools/` sandbox (`npx tsc --noEmit`) before ever promoting it to production.

### 3. API Protection
Any new HTTP endpoints added to `server.ts` must be protected by the global Rate Limiter to prevent DoS attacks.

### 4. Anti-Hallucination Standards
OctoArch is built for business environments. When tweaking LLM parameters or adding analytical roles, prioritize deterministic outputs (temperature: `0.1`) over creative guessing. The `CoderLLM` must always enforce strict JSON structuring and flawless TypeScript syntax.

### 5. Agnosticism & Multi-Tenant Isolation
The core of OctoArch should never depend on specific commercial ERPs. Keep specific business logic inside isolated MCP connectors or Contextual Markdown Skills. Furthermore, when altering database schemas in Supabase, you must ensure strict Multi-Tenant isolation using `client_id` and `session_id` to prevent data bleed between users.

### 6. Cloud Memory Management & Hot-Swapping
OctoArch is a stateful system backed by Supabase. If you implement features that consume RAM or modify chat history, ensure they sync properly with the cloud database. When contributing to the `DynamicRegistry`, ensure `delete require.cache` is used properly to prevent memory leaks when hot-reloading AI-generated `.ts` tools.

## 🚀 Pull Request (PR) Process

1. Fork the repository.
2. Create your feature branch (`git checkout -b feature/MyNewTool`).
3. Commit your changes with descriptive messages.
4. Test your changes locally ensuring the Cognitive Loop, Role-Based Access Control (RBAC), the Dynamic Registry (Hot-Swap), Supabase Memory retrieval, and Security Shields (`FileValidator`) are not broken.
5. Push to the branch (`git push origin feature/MyNewTool`).
6. Open a Pull Request detailing what problem your code solves and how to test it.

**Happy coding, let's build the future of autonomous systems together! 🐙**
